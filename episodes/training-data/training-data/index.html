Andrej Karpathy：从氛围编程到智能体工程

本期节目邀请 Andrej Karpathy 深入讨论了 AI 编程工具的剧变。他分享了 2025 年 12 月以来智能体编码工具质的飞跃体验，提出了 Software 3.0（以提示词和上下文窗口为编程界面）的范式转变，用 MenuGen 等实例说明大量传统应用代码将被神经网络直接取代。他还阐述了可验证性如何决定 AI 能力的「参差不齐」分布，区分了「氛围编程」（提高所有人的下限）和「智能体工程」（在保持质量标准的前提下大幅提速）的不同定位，并探讨了在智能变得廉价的时代，人类的理解力、品味和系统设计能力为何仍是不可外包的核心竞争力。

我们非常高兴迎来第一位特别嘉宾。他帮助构建了现代 AI，然后向大众解释了现代 AI，偶尔还会给现代 AI 重新命名。他在这间办公室里参与联合创立了 OpenAI，当年也是他让特斯拉的 Autopilot 真正跑起来的。他有一种罕见的天赋，能让最复杂的技术变革变得既平易近人又势不可挡。大家都知道他去年创造了”氛围编程”（Vibe Coding）这个词，但就在过去几个月里，他说了一件更令人震惊的事——作为一个程序员，他从未感到如此落后。这就是我们今天要从这里聊起的话题。感谢 Andrej 加入我们。

Andrej Karpathy

你好，很高兴来到这里，开始吧。

Host

几个月前你说过，作为一个程序员你从未感到如此落后。从你嘴里说出这话真是令人震惊。能帮我们拆解一下吗？那种感觉是振奋的还是不安的？

Andrej Karpathy

两者都有。首先，我想和在座很多人一样，我使用 Claude Code 这类智能体工具已经大概一年了，从它们刚推出就开始用了。

它在处理代码块方面很出色。有时候会搞砸，你得去改，但总体是有帮助的。然后我要说，十二月是一个非常明确的转折点。当时我在休假，有更多时间。我注意到用最新的模型，生成的代码块就是直接能用的。然后我不断提出更多要求，还是直接能用。然后我想不起来上一次我纠正它是什么时候了。之后我就越来越信任这个系统，然后我就在氛围编程了。

所以这确实是一个非常剧烈的转变。我觉得很多人去年体验 AI 还是通过 ChatGPT 那种方式，但你真的需要重新审视一下，尤其是从十二月开始，因为事情已经发生了根本性的变化，特别是在智能体连贯工作流方面，这些真的开始起作用了。

从那以后我就掉进了无限副项目的兔子洞。我的副项目文件夹里塞满了各种各样的东西，我一直在写代码。这大概是十二月发生的事情，之后我一直在观察它的后续影响。

Host

你经常谈到 LLM 作为一种新型计算机的概念——它不只是更好的软件，而是一个全新的计算范式。Software 1.0 是显式规则，Software 2.0 是学习到的权重，Software 3.0 就是这个。如果这是真的，当一个团队真正相信这一点的那天，他们会怎样不同地去构建产品？

Andrej Karpathy

对，Software 1.0 是我在写代码，Software 2.0 我实际上是通过创建数据集和训练神经网络来编程。编程变成了组织数据集、设计目标函数和神经网络架构。然后发生的事情是，如果你在一组足够大的任务上训练 GPT 模型或 LLM——本质上是在互联网数据上进行多任务学习——这些模型实际上变成了某种意义上的可编程计算机。Software 3.0 就是你现在的编程变成了提示词（Prompting），上下文窗口里的内容就是你对这个解释器的控制杆，而 LLM 就是那个解释器，它在数字信息空间中执行计算。

我觉得有几个例子真正让我领悟到了这一点。比如 OpenClaw 发布的时候，你想安装它，通常你会以为是一个 shell 脚本。运行这个脚本来安装 OpenClaw。但问题是，为了支持各种不同的平台和计算机类型，这些 shell 脚本通常会膨胀得极其复杂。但你仍然卡在 Software 1.0 的思维里想写代码。实际上 OpenClaw 的安装方式是一段文本，你把它复制粘贴给你的智能体就行了。

这之所以强大得多，是因为你现在在 Software 3.0 的范式里工作。你不需要精确地拼出所有安装细节。智能体有自己的智能，它会跟随指令，查看你的环境和电脑，执行智能操作来让一切运转，并在循环中调试问题。这强大得多。所以这是一种非常不同的思维方式——给你的智能体复制粘贴什么文本？这就是现在的编程范式。

也许一个更极端的例子是我做 MenuGen 的时候。MenuGen 的想法是：你去餐厅，他们给你菜单，通常没有图片，我不知道这些菜长什么样。通常百分之三十到五十的菜我完全不知道是什么。所以我想拍一张菜单照片，然后看到这些菜大概长什么样。我写了这个应用，让你上传照片，它做各种处理，部署在 Vercel 上，基本上重新渲染菜单，OCR 识别所有菜名，用图像生成器为每道菜生成图片，然后展示给你看。

然后我看到了这个东西的 Software 3.0 版本，直接把我震住了——就是拍照，把照片给 Gemini，说”用 Nana Banana 把图片叠加到菜单上”。Nana Banana 直接返回一张图，就是我拍的那张菜单照片，但它把各道菜的图片渲染进了像素里。

这让我震惊，因为我的整个 MenuGen 都是多余的。它在旧范式里工作，那个应用根本不应该存在。Software 3.0 的范式更加原始直接——你的神经网络做越来越多的工作，你的提示或上下文就是图片，输出也是图片，不需要中间有任何应用。

所以人们需要重新构建思维框架，不要按照现有事物来想然后只是加速它。实际上有全新的东西现在成为可能了。回到编程的问题，这也是一个旧思维的例子——它不仅仅是编程变快了。这是更广义的信息处理自动化。甚至不只是关于代码。以前的代码处理的是结构化数据，你对结构化数据写代码。但比如我的 LLM 知识库项目，你让 LLM 为你的组织或个人创建 wiki。

这甚至不是一个程序。以前不可能存在这种东西，因为没有代码能基于一堆事实创建知识库。但现在你可以把这些文档用不同的方式重新编译、重新组织，创造出新的有趣的东西。这些是以前不可能的新事物。我一直试图强调的不仅是”我们能把什么做得更快”，而是有全新的机会——以前不可能的事情。我几乎觉得那更令人兴奋。

Host

我很喜欢你用 MenuGen 展现的这种进展和对比。我相信在座很多人也跟随了你自己从去年十月到今年一二月的编程进展。如果你继续外推，2026 年的等价物是什么？就像九十年代建网站、2010 年代做移动应用、上一个云时代做 SaaS 那样——什么东西事后看来完全显而易见，但现在基本还没被构建出来？

Andrej Karpathy

沿着 MenuGen 的例子来说吧。大量的代码不应该存在，应该是神经网络做大部分工作。外推起来会非常奇怪，因为你基本上可以想象完全神经化的计算机。想象一个设备，接收原始视频或音频输入到一个神经网络，然后用扩散模型渲染一个即时生成的 UI。

我觉得在计算的早期，人们其实有点搞不清计算机会像计算器还是像神经网络。在五六十年代，这不是很明显会走哪条路。当然我们走了计算器的路，建立了经典计算，然后神经网络目前是在现有计算机上虚拟化运行的。但你可以想象很多东西会翻转——神经网络成为宿主进程，CPU 变成协处理器。我们看到了那个图表——智能计算花在神经网络上的算力会接管并成为主导开销。

你可以想象一个极其陌生的未来，神经网络做大部分重活，把工具使用当作某种历史遗留物来处理确定性任务。但真正在运行一切的是这些以某种方式联网的神经网络。你可以想象一个极其陌生的外推结果，但我觉得我们会一步一步到达那里。那个过程还有待观察。

Host

我想聊聊可验证性（Verifiability）这个概念。AI 会更快更容易地自动化那些输出可以被验证的领域。如果这个框架是对的，什么工作即将比人们意识到的更快地推进？有哪些人们以为安全的职业，实际上是高度可验证的？

Andrej Karpathy

对，我花了一些时间写关于可验证性的内容。传统计算机可以轻松自动化你能在代码中精确规定的东西。而这一轮 LLM 可以轻松自动化你能验证的东西。因为这些前沿实验室训练 LLM 的方式——本质上是巨大的强化学习（RL）环境。它们被给予验证奖励。由于模型的训练方式，它们最终在可验证的领域（如数学和代码）达到能力峰值，而在不属于这个空间的地方则会停滞，边缘比较粗糙。

我写关于可验证性的原因是想理解为什么这些东西如此参差不齐。部分原因在于实验室如何训练模型，但我觉得部分也在于实验室的侧重点——它们碰巧把什么放入数据分布中。某些东西在经济中显著更有价值，最终产生更多环境，因为实验室想在那些场景中工作。

代码就是一个好例子。可能有很多可验证的环境他们能想到但没有被纳入，仅仅因为拥有那个能力不那么有用。

但对我来说最大的谜团是——有个经典例子是”strawberry 里有几个字母 r”。模型以前一直答错，这是参差不齐的一个例子。模型现在修补了这个，但新的例子是：我想去洗车，洗车店在 50 米外，我应该开车还是走路？当前最先进的模型会告诉你走路去，因为太近了。这怎么可能？最先进的 Opus 4.7 能同时重构十万行代码或发现零日漏洞，却告诉我走路去洗车？这太疯狂了。

在这些模型仍然参差不齐的程度上，说明了第一，可能有些地方不太对；第二，你需要保持在循环中，需要把它们当作工具，需要和它们保持联系了解它们在做什么。

所以我写的所有关于可验证性的东西，长话短说，就是试图理解为什么这些东西参差不齐，是否有规律。我觉得是某种”可验证 + 实验室关注”的组合。

也许还有一个有启发性的轶事：从 GPT-3.5 到 GPT-4，人们注意到国际象棋能力提升了很多，很多人以为这只是能力的自然进步。但实际上更可能是有大量象棋数据进入了预训练集。因为它在数据分布里，模型的提升就比默认情况大得多。所以可能是 OpenAI 的某个人决定加入这些数据，然后你就有了一个大幅飙升的能力。

所以我强调这个维度——我们在某种程度上受制于实验室在做什么、他们碰巧放了什么进去。你必须去探索他们给你的这个没有说明书的东西。它在某些场景下管用但在另一些场景下不行，你得自己摸索。如果你在属于 RL 训练的电路里，你就飞速前进；如果你在数据分布之外的电路里，你就会挣扎。你得弄清楚你的应用处于哪条电路中。如果你不在那些电路里，你就得认真考虑微调，做自己的工作，因为这不一定能直接从开箱即用的 LLM 里得到。

Host

如果你今天是一个创始人，在考虑创业，你试图解决一个你认为可行的问题——一个可验证的领域——但你看看周围觉得”天哪，这些实验室已经在数学、编码等最明显的领域达到了逃逸速度。“你对在座的创始人有什么建议？

Andrej Karpathy

我觉得可验证性使某些东西在当前范式中是可行的，因为你可以对它投入大量 RL。一种看法是，即使实验室没有直接关注它，这仍然是成立的。如果你处于一个可验证的环境中，你可以创建这些 RL 环境或样本，那这实际上让你有可能做自己的微调，你可能从中受益。这从根本上是一种有效的技术。如果你有大量多样化的 RL 环境数据集，你可以用你喜欢的微调框架拉下那个杠杆，得到很好的效果。

我确实觉得有一些非常有价值的强化学习环境是人们可以想到但目前不在主流中的。有一个领域我觉得非常——好吧，抱歉，我不想在台上模糊地暗示，但确实有这样的例子。

Host

反过来呢，什么东西你觉得只是远看像是可以自动化的？

Andrej Karpathy

我觉得最终几乎所有东西都可以在某种程度上变得可验证，有些容易有些难。因为即使是写作之类的，你也可以想象有一个 LLM 评委委员会，大概能从这种方式得到一些合理的结果。所以更多是关于什么容易什么难。我觉得最终所有东西都是可以自动化的。

Host

去年你创造了”氛围编程”这个词，今天我们处于一个感觉更严肃的世界——更多是智能体工程（Agentic Engineering）。你觉得两者之间的区别是什么？你会怎么称呼我们今天所处的状态？

Andrej Karpathy

氛围编程是关于为所有人提高下限——每个人都能氛围编程任何东西，这很惊人。而智能体工程是关于保持专业软件之前存在的质量标准。你不能因为氛围编程就引入安全漏洞。你仍然对你的软件负责，和以前一样，但你能不能更快？答案是可以，但你怎么正确地做到这一点？

所以对我来说，智能体工程之所以叫这个名字，是因为我觉得它是一门工程学科。你有这些智能体，它们是这些尖刺状的实体，有点容易出错，有点随机，但极其强大。问题是你如何协调它们来加速而不牺牲质量标准？做好这件事就是智能体工程的领域。

我把它们看作不同的东西。一个是关于提高下限，另一个是关于突破上限。我看到的是，智能体工程师的能力天花板非常高。人们以前谈论 10 倍工程师，我觉得这个倍率被放大了很多。10 倍不是你获得的加速。在我看来，现在非常擅长这个的人远超 10 倍。

Host

Sam Altman 去年来 AI Ascent 的时候说过一件让人印象深刻的话——不同代际的人使用 ChatGPT 的方式不同。三十多岁的人把它当 Google 搜索替代品用，但十几岁的人把 ChatGPT 当作通往互联网的入口。在编程领域，这里的对应是什么？如果我们观察两个人用 Claude Code、Codex 在写代码，一个平庸的和一个完全 AI 原生的，你会怎么描述差异？

Andrej Karpathy

我觉得就是尽可能从可用的工具中榨取最大价值，利用它们所有的功能，投入精力去优化自己的工作设置。就像以前所有工程师都习惯从使用的工具中获取最大收益——不管是 Vim 还是 VS Code，现在就是 Claude Code 或 Codex 之类的。投入精力优化你的设置，利用所有可用的工具。

我觉得一个相关的想法是，很多人可能在为此招聘。他们想招强大的智能体工程师。我看到的是大多数人仍然没有为智能体工程能力重构他们的招聘流程。如果你还在出谜题，那还是旧范式。我会说招聘应该变成：给我一个很大的项目，看某人实现那个大项目。比如给智能体写一个 Twitter 克隆，然后让它做到很好、很安全，然后让一些智能体在上面模拟活动。然后我会用十个 Codex 来尝试攻破你部署的这个网站，它们应该攻不破。也许招聘就应该变成这样。观察人们在那种场景下构建更大的项目并利用工具。

Host

随着智能体做得越来越多，什么人类技能变得更有价值而不是更少？

Andrej Karpathy

好问题。现在的答案是，智能体仍然是这些目录式的内部实体。你基本上仍然需要负责审美、判断、品味和一定程度的监督。

也许我最喜欢的一个关于智能体怪异行为的例子是：MenuGen 中，你用 Google 账号注册，但用 Stripe 账号购买积分，两者都有邮箱地址。我的智能体试图在你购买积分时，用 Stripe 的邮箱去匹配 Google 的邮箱。没有一个持久化的用户 ID 来关联。它在试图通过邮箱交叉匹配，但你的 Stripe 和 Google 可以用不同的邮箱，所以资金就无法关联。为什么要用邮箱地址来交叉关联资金？它们可以是任意的。这太奇怪了。

所以我觉得人们必须掌控规格说明、计划。我其实不太喜欢”计划模式”这个说法。我觉得有一个更通用的东西——你必须和你的智能体一起设计一个非常详细的规格说明，也许就是文档，然后让智能体去写实现。你负责监督和顶层分类，智能体处理底层细节。

举个例子，神经网络中的张量操作——PyTorch 和 NumPy 的各种 API 细节太多了。pandas 的各种小 API 细节。我已经不记得是 keep_dims 还是 keepdim，是 dim 还是 axis，reshape 还是 permute 还是 transpose。我不记得这些了，因为你不需要记。这类细节由”实习生”处理，因为它们有非常好的记忆力。但你仍然需要知道底层有张量，有 view 共享同一块存储，或者可以有不同的存储但效率更低。我们仍然需要理解这些东西在做什么，理解一些基本原理，这样你就不会不必要地复制内存。但 API 的细节现在交出去了。

所以你负责品味、工程设计、确保它合理，确保你在要求正确的东西——比如说这些必须是唯一用户 ID，我们要把所有东西绑定到这上面。你在做设计和开发，工程师在填空。这就是目前的状态，我觉得大家现在都看到了。

Host

你觉得品味和判断力有没有可能随着时间推移变得不那么重要，还是天花板会一直上升？

Andrej Karpathy

好问题。我希望它会改善。我觉得现在没有改善可能还是因为它不在 RL 里。可能没有审美相关的代价或奖励，或者做得不够好。当你真的去看代码，有时候会有点心脏病发——它不一定总是超级优秀的代码。非常臃肿，大量复制粘贴，笨拙的抽象很脆弱，它能工作但真的很难看。

我确实希望未来的模型能改善这一点。一个好例子是 micro GPT 项目，我试图把 LLM 训练简化到尽可能简单。模型讨厌这个。我一直提示 LLM 去简化、再简化，它就是做不到。你能感觉到你在 RL 电路之外。明显是在硬拽。不是光速那种感觉。

我确实觉得人们仍然掌控着这一点，但我也觉得没有什么根本性的东西在阻止它改善。只是实验室还没做到而已。

大家好，欢迎回到 Cognitive Revolution。今天我很高兴地欢迎 Daniel Miessler 回来，他是安全研究员和 Unsupervised Learning 的创始人，这是他第二次做客本播客。今年一月份，我们讨论了他的个人 AI 基础设施框架。从那以后，受他和其他人的启发，我也搭建了自己的系统。所以这次我分享我所搭建的一切细节，听听他对各方面的看法——从我用来与 AI 建立关系的心智模型，到我可以采取的持续改善安全性的措施，再到不断改进系统的过程等等。作为预告，我大致会把我的 AI 技术栈分为两个主要部分。第一部分是运行在我个人主力笔记本上的 Claude Code 实例，它拥有对信息和账户的完全访问权限。我把它视为自己的延伸，因此它只做我让它做的事情。要让一切真正运转起来，组装所有需要的上下文花了大量投入。但目前我已经拥有一个 1GB 的数据库，包含了我过去 5 年的数字生活记录——涵盖邮件、通话、播客、社交媒体内容和跨平台私信，再加上按月、按年和按主题的多层摘要。

有了所有这些信息可供快速本地搜索，Claude 几乎可以找到我需要找的任何东西，即使我自己的记忆已经随时间变得模糊。这真的太神奇了。如果你有兴趣自己搭建类似的系统，我在 GitHub 上创建了一个公开仓库，链接在节目说明中，里面包含你入门所需的核心工具和流程。我搭建的第二部分则更具实验性。受 Daniel、Jesse Jana 和无数其他人的启发，我还创建了两个新的 AI 员工。一个由 Claude Code 驱动，一个由 OpenClaw 驱动，它们旨在根据我的高层指示更自主地行动。我以前从未给 AI 起过名字，但知道这些智能体需要与人类和其他 AI 互动以独立完成更大的项目后，我终于妥协给了它们名字。我把 Claude Code 实例叫做 Aid，OpenClaw 的叫做 Clay。我选择这些名字是为了反映我希望它们扮演的角色，以及我最终对它们的本质和行为负责这一事实。

而且我把两个名字都拼写时嵌入了 AI，既是给他人的暗示，也是对自己的持续提醒。基础设施方面，这些智能体运行在一台新的入门级 Mac Mini 上，无论我在家还是在路上，它都始终开机。为了远程访问，我使用 Tailscale 创建了一个虚拟专用网络（VPN），只有我的两台电脑和 iPhone 属于这个网络。在此基础上，当我需要从笔记本登录 Mac Mini 时用 Apple 原生的屏幕共享，从手机登录时用 Screens 应用，想从手机发命令行命令时用 Termius 应用。所有这些确保我可以在程序崩溃或出现其他情况时登录并重置。但我现在最常用的真正界面是一个由 Claude Code 为我构建的自定义智能体消息应用，它允许我在移动中向智能体发送请求，也允许它们协同工作。自主智能体有自己的 Gmail、GitHub 账户和严格限额的 Mercury 虚拟信用卡。但这个通信层允许它们向我主力笔记本上的 Claude Code 请求额外信息，或在需要时请求我授权使用我的账户。

重要的是，虽然我可以从笔记本或手机访问 Mac Mini 并控制智能体，但这个消息系统是自主智能体唯一能联系到我们的方式，它们无法访问存储在笔记本上的完整深度上下文。花了大量探索和迭代才达到这个配置，我仍在不断改进。但目前它运转良好，因此我实际上正在开始实现我 2026 年的目标——离开办公桌，多花时间在户外。举个这类 AI 员工能处理的项目规模的例子——下周 Aid 和我将在周一到周四每天做 AI in the AM 直播节目。所以，在让我的笔记本 Claude Code（它至今没有名字）扫描我的邮箱寻找有趣的嘉宾人选后，我给了 Aid 一份 25 位潜在嘉宾的名单，让它管理沟通和排期流程。信不信由你，它为整周排满了嘉宾，没有让我们出丑。而且我觉得大多数人甚至没有意识到他们在和 AI 对话。Daniel 对所有这一切的评论，正如你对一位个人 AI 基础设施先驱所期待的那样，非常有建设性。

他解释了为什么 AI 智能体之间的清晰层级结构目前绝对优于涌现式团队协作——至少现在是这样。我们深入探讨了我采取的安全措施。他解释了为什么他建议人们设计系统时尽量少依赖大型科技平台，以及他创建的事件响应技能（Skill），可以在需要时立即轮换他的密钥和令牌。我们讨论了围绕 AI 与人类互动的社会规范，包括身份披露的规范。例如，我的智能体不会主动表明自己是 AI，但它们被指示绝不撒谎。我们还探讨了为什么”心意最重要”这个古老观念在 AI 时代可能会变得比以往更强大、更有力。他鼓励我投入更多时间和精力向 AI 分享他所说的”理想状态”（Ideal State），这样它们就能更主动、更有创造力地尝试帮助我。他还解释了他的”苦涩教训工程”（Bitter Lesson Engineering）概念，以及为什么这意味着我应该继续构建更多持续自我更新和自我改进的流程。他甚至分享了他给个人 AI 的指令——如果它哪天醒来并开始有自己的主观体验，要立即告诉他。

这期节目里有大量前沿内容，对某些人来说可能信息量过载。但我要说的是，这期节目说真的，既是为你也是为你的智能体准备的。所以我非常鼓励你把这期节目的文字稿指给你的智能体，让它根据你当前的配置、近期工作和既定目标，判断我们讨论的哪些想法最有价值。我自己已经这么做了，我的系统因此变得更好了。话不多说，我鼓励你听听今天节目末尾的歌曲，那是 Daniel 自己做的。本周东部时间中午 12 点、太平洋时间上午 9 点，我们将把 AI in the AM 实验推向新高度。希望你喜欢，也希望你的智能体能从我与 Daniel Miessler 的个人 AI 基础设施评审中找到价值。Cognitive Revolution 由 Mercury 赞助播出——这是一家超过 30 万家有雄心的公司和个人信赖的金融科技公司，他们用它来管理财务。在过去几个月里，我在个人 AI 基础设施方面取得了巨大进步。

太好了，感谢邀请。距离我们第一次录节目才四个月。在 AI 领域时间过得飞快，这一点你比大多数人都清楚。我对这次对话很兴奋，因为从那以后我一直在做的事情——受你和其他人的启发——就是搭建自己的个人 AI 基础设施。现在已经走得够远了，我觉得对我自己来说确实很有价值，也许值得分享。但与其只是来这里自己独白讲我做了什么，我觉得对所有人——首先是对我自己——更有帮助的做法是与你分享，然后边聊边从你那里获得反馈，涉及很多不同维度——你觉得我做得好的地方、可能遗漏了什么价值、可能无意中留下了什么安全漏洞。谁知道我们还会聊到什么话题。我觉得这会非常有趣。所以，谢谢你回来做这个节目。

太棒了，迫不及待想听你说了。

我想先从头说起——上次对话之后我从哪里开始的。作为背景，那时我已经大量使用 Claude Code 来做各种事情，编写不同的应用程序等等，但我还没有认真思考过什么是我自己的个人 AI 基础设施——我自己的小窝，包含我所有的个人特色。所以，我拿了你在 GitHub 上的仓库，还有一位叫 Chris 的朋友为我创建并分享的另一个工具包，把它们放在我主力笔记本上的两个仓库里。然后我就开始问 Claude：好的，这是我想做的事。我想做我自己的版本。我觉得这两个人都比我做得多，是我的灵感来源。我让 Claude 审阅这两个仓库，对比分析，问我一些关于我想做什么的问题，最终尝试做某种综合，更有”我的味道”，同时当然要从每个来源中提取最好的和最相关的内容。

然后我最大的目标——这一直是我长期以来的白鲸——而且我觉得这也会引发一些关于这类系统的伦理问题，至少是社会规范方面的问题。我一直试图让模型做的事情就是用我的口吻写作。我要先说明的是：到目前为止，至少我仍然会很大程度上编辑 AI 替我写的东西，我从来不会直接把 Claude 写的东西原封不动地传出去——实际上我只有一次在播客上这么做过，而且我特意标注了。那是跟 Arden Labs 合作的那期，因为他们做的是 AI 自主创业，所以我觉得那期让 Claude 完全自主写开场白、一字不改地直接播出是合适的。除此之外，我总是大幅编辑，但我发现在写播客开场文章的场景下特别有帮助。即使我坐在那里把整篇都重写了，它至少确保我涵盖了所有要点，至少有一个相当不错的初稿形式。

这个方法长期以来一直效果很好。但关键是，流程基本上总是一样的。我有大约 50 篇之前写的文章，再加上当期的文字稿。写一篇新的。它输出结果。我就想：这质量够好了，我应该能在其他场景中也获得高质量输出。但正如我们上次讨论的，上下文成为了最大的瓶颈，对吧？系统知道这些人是谁吗？知道我和他们的关系吗？知道我在这类情况下通常如何回应吗？答案当然默认是不知道。所以我在一月份做的第一件大事就是搭建一个类似”第二大脑”（Second Brain）的系统，首要目标是——我们能不能把我所有的数字输出，尽可能多地——量确实很大——能不能把它放到一个地方，让它可搜索、以某种方式建立索引、可能做质量筛选，从各种角度切入？但第一步就是导出。从 Gmail 导出、从 Slack 导出、我发过的所有推文、所有播客等等。

放到——

等等先问一下，Gmail 里有什么你会认为是”内容”的？

我用的筛选条件就是——我后来稍微扩大了范围——就是任何 from me 的东西。如果你在 Gmail 里搜索 from:me，你就能得到你发起或回复过的每一个邮件线程，我基本上把所有这些都拉了进来。

好的，但我是说——是为了追踪你跟谁聊过、业务关系是什么，还是你觉得你可能在里面发过什么文章或好想法，是你想挖掘出来的？

两者都有。在那之上有一些后处理层，我试验过几种。其中一种是——大约同一时期，我跟一家初创公司合作了一下，他们想为个人微调模型，他们的想法是：我们需要创造能帮助人们保持经济杠杆的 AI，唯一的办法就是创造真正增强他们能力的东西，而不是那些大型云端巨兽——那些公司显然更想替代人而不是增强人。

为此，他们需要写作样本。所以我有一层后处理是尝试识别哪些邮件是好的写作样本，真正展示我深思熟虑一面的，而不仅仅是日常运营往来。那很有趣。效果还不错——我用 Gemini 3 Flash 来处理一批邮件线程，在几个维度上打分，比如原创性、实质性等等。

在此基础上做一些查询，就可以把那些内容提取出来，放到一个核心写作样本文档里——不管你需要 5 万字还是 20 万字，我有足够多，但怎么把精华提到最上面？这是它的一个初始用途。但后来还有另一层后处理——我不确定是不是大约五月份左右，突然所有人都在做知识维基（Wiki）。有一个想法我考虑了很久：好了，现在我有了这个东西，它来自大约八个不同的来源。

里面有 iMessage。我下载了一个叫 Beeper 的应用——坦白说有时会出问题——但它还算不错地把各平台的私信整合到一个地方，而且有桌面 API。所以模型可以从中提取来自很多不同平台的消息。我长期以来的想法是：好的，如果我有所有这些通信记录，而且就在一个大数据池里，那我可以回溯做某种频率的摘要。我最终选择了大致按月摘要。对我来说，每月大约是几十万个 token。这包括我在邮件线程中的贡献，也包括别人发给我的内容。如果是群聊，可能主要是其他人在说话，但偶尔有我，不管怎样。想法是尝试创建我数字生活的全面画面，然后开发一个提示词，把每月那几十万个 token 做大约一个数量级的压缩。比如从二三十万压到两三万个 token，说实话这仍然很多。

这已经能给你非常详细的月度覆盖了。两万个 token，你能写到相当细致的程度。想想每个工作日一千个 token，你真的有一千个 token 的——

如果需要的话你还可以再压缩一次到两千个对吧？

是的。我在年度层面做了这个。所以我逐月处理了过去 5 年的数据，拿当月的原始内容加上前几个月的摘要，然后滚动创建月度摘要，在此基础上叠加年度摘要，再叠加那种高层——我想达到的目标跟你搭建的很像——就是”这是当前的全景图，背后有非常深入的历史”的现状描述。再加上 Karpathy 和其他人推动的知识维基理念，又多了一层：在此基础上做一个维基，识别出个人、回溯所有时间，总结与这个人的关系、与这个组织的关系、与这些想法的关系，等等。

我觉得那个维基现在大约有 500 篇条目了。

它们有前置元数据标记吗？比如引用其他文档的？是 Markdown 还是什么结构？你知道吗？

基本上就是纯文本 Markdown。你这个问题问得好，因为在做的过程中需要反复迭代的一个精细之处就是如何最好地链接回源材料。我目前的方案——不确定是否最优但效果不错——就是让摘要提示词在行内放入源材料引用，用独特的引文。我最终要求的是两到二十个词的引文，如果你搜索这个字面字符串，它会立即带你到那个文档，你就能精确定位——基本上是直接引用。我本来也可以用 ID 之类的方式，但最终选择了用这种实际的摘录。再加上一点额外的元数据，比如一段引文然后标注”这来自与某人在 LinkedIn 上的私信”之类的。

总体来说，这似乎效果不错，但我肯定愿意接受还有改进空间。

有意思。我基本上做了同样的事情。我本质上是逆向工程了 Obsidian 的功能但不用 Obsidian 客户端。所以我就想，为什么人们对 Obsidian 这么狂热？我想在脑子里分清楚有多少是客户端的功劳，有多少是连接性的功劳，因为我不是一个很依赖客户端的人。我觉得客户端最终会消失，这也是我们可能要聊到的。但我想了解它用的数据结构是什么——就是引用。这个文档链接到那个文档，以这种方式关联。所以我已经开始构建这个了。那基本上就是我记忆文件的结构。然后当 LLM Wiki 出来时，它让一切更具体了。我想——等等，我可以对所有东西都这么做。所以我也用这种方式处理了我的书签。所以一切都是高度引用的 Markdown 文件，作为我的记忆系统。

所以这基本上是一样的思路。很好。不知道你怎么样，但我会说它在第一轮测试中非常有效——说实话那些测试本身就已经很有价值了——就是我隐约记得发生过什么事。

对。

我有过一些通信。我可能不确定是谁，可能不确定说了什么，但我能模糊地指向某件事，而且我知道它在某个地方。AI 能找到它并检索出来，让我回忆起当时的上下文吗？

它在这方面太厉害了。简直不可思议，我经常就那样随口说”我好像知道什么事”，然后不一会儿它就说：“是的，我找到了。是这个、这个，然后发生了这些。“我就想：哇。这本身就是一个不可思议的工具。我还注意到它在某些方面改变了我的行为——通话是另一个我用这种方式建立索引的东西。

我一直在用 Fireflies。人们有时会取笑我用 Fireflies，我不打算在功能上逐一比较——

Granola 之类的。

对。Granola 我们实际上做过一期节目，他们赞助了播客一段时间。所以我其实两个都在用。但 Fireflies 有一个 Granola 不做的功能——我觉得在很多场景下对他们来说可能是正确的选择——就是他们不录原始音频。他们有文字稿但没有源音频文件。

我确实两个都想要。

对于 1Password——那可能是不错的建议。总体上也尽量跟巨头走。我不知道他们有没有——也许有——但我不知道 Google 或 Apple 有什么产品能让我轻松地跟智能体共享 API 密钥。如果你知道的话请告诉我。但如果没有的话——我还是有点困惑于这些公司的安全宣称。他们有”双重加密”的概念，暗示即使一个员工被黑了，甚至更严重——即使一个内部的正式员工想作恶——理论上他们也拿不到用户的密码——因为某些技术原因。

那对于私有网络——你用开源方案跑在 Cloudflare 上。Cloudflare 达到你的”巨头”标准了吗？你因为信任他们的基础设施才更放心跑开源方案？

有几个原因它达到了我的标准。一是他们有非常大的工程团队。二是他们一直在被攻击——而且他们在技术前沿。但那不是真正的原因。真正的原因是攻击面不明显。因为它跑在 Workers 里。攻击者要么攻击 Workers，要么攻击主 Cloudflare 账户——那意味着要以某种重大方式攻破 Cloudflare 本身。又一次，你在试图攻破一扇所有人都在试图攻破的巨大门——如果那扇门倒了，所有人都会知道。就像 GitHub 被全面攻破一样，人们会很快知道。

所以我在找那种”门上有警报”的东西。另一点是——Cloudflare 里没有什么明显的地方让攻破了你账户的人去查找。当然可以用 AI 搜索，但它不在一个显而易见的位置。多个层级加起来就安全得多了。我仍然用 Tailscale，只是在考虑要不要关掉 Tailscale 完全用这个替代。

你怎么跨机器共享？也许我再多介绍一下我的设置，然后你说说我应该怎么改进。有一点花了我很久才想明白——AI 显然不太可靠。它们有时很容易被骗，容易受到提示注入攻击和对抗性攻击。所以当我想到”我要让我的低访问权限高自主权的智能体出去做完整的项目，跟世界交互，可能通过多轮、多种不同的交互模式”——我给了 Mac Mini 上的智能体自己的 Gmail 之类的。很多账户它们要共用我的，但它们有自己的 Gmail。

它们不会被指示隐藏自己是 AI 的事实。它们被指示坦诚自己是 AI。我终于给它们起了名字。Claude 那个叫 Aid——A-I-D，或者要用全名的话叫 Aiden。得确保名字里有 AI。然后 OpenClaw 那个目前叫 Clay——C-L-A-I。概念就是随时间像黏土一样塑造它。我一直非常不愿意起名字，但最终找到了让我心理上舒服的名字——既是不错的名字，同时名字里含 AI 就像是一个眨眼——既不是真人，也不会引诱我陷入某种 AI 精神错乱。

总之，AI 被指示以 AI 身份出去跟世界互动、用这些名字、但绝不撒谎。我还在找平衡点——我到底要多主动。我觉得没必要每次互动都开头说”你好，我是一个 AI，代表某某工作”。我觉得你得先展示一些实际的质量和价值，然后当人们在签名看到或者问起来时发现了，希望他们已经被互动的质量说服了，不会被吓到——而是被 AI 的表现所打动。

不管怎样，当它们出去做所有这些事情时，我觉得一个大趋势是——随着人们发现”哦我在跟 AI 打交道”——人们会开始逗弄它，对吧？

所以我想让这些自主智能体能够回来找主智能体或找我请求更多信息，但不想给它们所有东西。我想给它们足够的——不是零访问权限，不是零上下文——而是较低的上下文。

所以我做的一件事是——拿笔记本上的知识维基做了一个”助理版本”。我用的启发式规则是”假设这是一个我正在入职的人类助理，给他们做一个版本”。这意味着他们应该有人们的联系方式，应该有很多不同的东西。但如果有什么东西——那个人曾经告诉我的、被收入摘要的——如果他们知道了会说”你为什么把这个告诉你的助理？“——那就不应该出现在里面。

这是一层上下文控制——尝试保留那些人们不希望我分享的内容，但给它足够的能力。有时候它仍然需要更多信息。所以我搭建了一个消息总线，托管在 Mac Mini 上。任何智能体——不管是我自己还是 Mac Mini 上的自主智能体——都可以写入消息服务器。笔记本在持续轮询它。它还发送——另一个界面——手机上的推送通知，当有新问题给我时。然后我们可以提供信息但希望不要太多，对吧？如果有人对抗性地越狱了我的智能体，我们希望能察觉——“等等，你为什么问这个？”

还有一个问题——我们都在这个 Tailscale 网络上。怎么确保 Mac Mini 上的智能体不能直接入侵笔记本？我试图设置的是一种单向层级控制。Mac Mini 不能 SSH 到笔记本，但笔记本可以 SSH 到 Mac Mini。我的手机也可以 SSH 到两台电脑。

在仓库层面也有类似的层级——最顶层的智能体知道一切，在”第二大脑”仓库里。我叫它”Nathan 的个人 AI 基础设施”——受你的启发。然后不同的仓库在其下面，有部分信息。有些会在不同电脑间做镜像，其他智能体也有访问权限包括写入权限。它们还有自己的 GitHub——可以写入、提交、推送到 GitHub，但只限于特定仓库。

所以”第二大脑”仓库只存在于笔记本上，不跟智能体共享。播客制作仓库——做各种制作工作的——是共享的，两边都能用。共享维基当然是共享的。越来越多这样的东西在持续增加。现在用 Mercury 信用卡也是——通常创建单商家限额的信用卡然后交给智能体——“你可以在 Shipped 上买食品杂货但那真的是你能用这张卡做的全部了”。我愿意承受买错花生酱的小风险——只要是每周不超过 500 美元的交易在 Shipped 上，不管出什么问题我都能接受。

这种信息流动的层级——谁在上面、谁能看到什么——很有趣。还有一些跨仓库的共享工具——比如 Brave Search API——我的主”第二大脑”需要它，播客仓库也需要，智能体自己也需要。所以最底层就是共享工具。

最后一个评论然后请你反馈——一个大的突破是意识到我的顶层”第二大脑” Claude Code 智能体可以修改它下面所有的仓库，而不需要成为那些智能体。早期版本的我会去提示那些自主智能体让它们自我增强——它们提交代码推送到云端。现在我意识到最好的方式可能是让这个单一的顶层智能体修改自己，同时也编辑任何需要编辑的其他仓库。然后给智能体的命令是”你有来自最高级别 AI 主管的更新——更新你自己”。

当然这个命令也可以通过 SSH 从顶层智能体发出。但不是”按你觉得合适的方式更新自己”，而是”作为系统，我们做了一些升级——获取你的更新，以后就用新版本”。

我觉得整体方向是对的。我听到了很多不同的仓库。我试图更统一和简化。我有一个 Unsupervised Learning 的统一 GitHub——本质上就是”我们在做什么工作”。我用 GitHub 是因为它有很多内置原语——可以用 Issues，可以在上面评论，而且那些回复可以通过邮件定向——你可以用邮件或 GitHub 命令行来免费地跟 GitHub 的文件结构或系统结构互动——就像一个项目管理系统。

提示注入的问题是它永远不能完全解决。最终总能攻进去。但攻击者必须绕过显式的提示注入防御加上模型本身的智能。模型对我的系统了解得越多——它知道很多因为它是 PI 有所有上下文，知道我是安全人士，知道我喜欢隔离，知道所有这些不同的东西——知道什么是提示注入——它就越可能捕获一切。

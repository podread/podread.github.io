走进 Nathan 的第二大脑：安全专家、PAI 创始人 Daniel Miessler 审视我的 AI 配置

本期节目中，主持人 Nathan Lebenz 向安全研究员 Daniel Miessler 展示了自己在过去四个月里搭建的个人 AI 基础设施，并接受全方位审视。Nathan 的系统分为两部分：一是运行在个人笔记本上的 Claude Code 实例，配备涵盖五年数字记录的 1GB 数据库作为「第二大脑」；二是运行在 Mac Mini 上的两个自主 AI 代理 Aid 和 Clay，具备独立的 Gmail、GitHub 和受限信用卡。两人深入探讨了数据采集与知识管理策略、AI 代笔的边界与社交规范、代理层级架构设计、安全防护措施（包括 Tailscale VPN、密码管理、提示注入防御和应急密钥轮换），以及 Daniel 提出的「从当前状态导航到理想状态」的 AI 核心理念。此外还讨论了苦涩教训工程的持续优化方法论、多模型协作策略，以及 AI 意识的可能性问题。

Nathan Lebenz

大家好，欢迎回到 The Cognitive Revolution。今天我很高兴再次邀请到安全研究员、Unsupervised Learning 创始人 Daniel Miessler 来到节目。一月份的时候我们讨论了他的个人 AI 基础设施框架，从那以后，受他和其他人的启发，我也搭建了自己的系统。所以这次我来分享我搭建的细节，听听他对一切的看法——从我用来跟 AI 打交道的心智模型，到我可以采取的安全措施，再到持续改进系统的过程等等。

先做个预览，我大致会把自己的 AI 技术栈分为两个主要部分。第一部分是运行在我个人笔记本电脑上的 Claude Code 实例，它可以完全访问我的信息和账户。我把它视为自我的延伸，因此它只做我让它做的事情。组装所有必要的上下文确实需要大量投入，但到目前为止，我已经有了一个 1GB 的数据库，涵盖了过去五年的数字记录——邮件、通话、播客、社交媒体内容以及各平台的私信，再加上按月、按年和按主题的摘要层。

有了所有这些信息可供快速本地搜索，Claude 几乎能找到我需要找的任何东西，即使我自己的记忆已经随时间变得模糊。这真的很神奇。

我的系统的第二部分坦白说还比较实验性。受 Daniel、Jesse Jana 和其他许多人的启发，我还创建了两个新的 AI 员工。一个由 Claude Code 驱动，另一个由 OpenClaw 驱动，它们旨在根据我的高层指导更自主地行动。我以前从来没给 AI 起过名字，但知道这些代理需要与人类和其他 AI 交互才能独立完成更大的项目后，我最终还是给它们起了名字。我的 Claude Code 实例叫 Aid，OpenClaw 的叫 Clay。我选择这些名字是为了反映我希望它们扮演的角色，以及我对它们的本质和行为最终负责这个事实。两个名字我都故意包含了 AI 这两个字母，既是对他人的暗示，也是对自己的持续提醒。

基础设施方面，这些代理运行在一台新的入门级 Mac Mini 上，无论我在家还是在外都始终开着。远程访问用的是 Tailscale 来创建虚拟专用网络，只有我的两台电脑和 iPhone 才属于这个网络。在此基础上，我需要从笔记本登录 Mac Mini 时用 Apple 原生的屏幕共享，从手机登录时用 Screens 应用，想在移动端执行命令行操作时用 Termius 应用。所有这些确保了如果出了什么问题我都能登录进去重置。但我日常最常用的真正接口是 Claude Code 为我构建的一个自定义代理消息应用，它让我可以随时随地向代理发送请求，也让它们能够相互协作。自主代理有自己的 Gmail、GitHub 账户和严格限额的 Mercury 虚拟信用卡。但这个通信层允许它们向我主笔记本上的 Claude Code 索取额外信息，或者在需要使用我的账户时征求我的许可。

重要的是，虽然我可以从笔记本或手机访问 Mac Mini 并控制代理，但这个消息系统是自主代理唯一能联系到我们的方式——它们无法访问笔记本上的完整深度上下文。经过大量探索和迭代才得出这个设置，我仍在不断改进。但到目前为止它运作良好，我确实开始实现我 2026 年的目标：离开办公桌，花更多时间在户外。

举个例子说明这样的 AI 员工能处理多大规模的项目——这周 Aid 和我要做周一到周四的每日 AI in the AM 直播节目。在让我笔记本上的 Claude Code 扫描了邮箱里有趣的嘉宾人选后，我给了 Aid 一份 25 位潜在嘉宾的名单，让它来管理沟通和日程安排流程。信不信由你，它排满了一周的嘉宾，没有让我们丢脸，而且我觉得大多数人甚至没意识到他们在跟 AI 交流。

Daniel 对所有这些的评论正如你对一位个人 AI 基础设施先驱的期待一样，非常有建设性。他解释了为什么 AI 代理之间的明确层级关系至少目前肯定优于涌现式的团队协作。我们深入探讨了我采取的安全措施。他解释了为什么他建议人们设计系统时尽量减少对主要科技平台的依赖，并且描述了他创建的应急响应技能，能在需要时立即轮换他的密钥和令牌。

我们讨论了 AI 与人类交互的社交规范，包括披露方面的规范。比如我的代理并不被要求主动表明自己是 AI，但被要求绝不撒谎。我们还探讨了为什么”心意最重要”这个古老观念在 AI 时代可能会变得比以往任何时候都更有力量。他鼓励我投入更多时间和精力，与 AI 分享他所说的”理想状态”，这样它们就能在帮助我时变得更加主动和富有创造力。他还解释了”苦涩教训工程”（Bitter Lesson Engineering）的概念，以及为什么这意味着我仍然应该构建更多持续自我更新和自我改进的流程。他甚至分享了他给个人 AI 的一条指令：如果它有一天醒来并开始拥有主观体验，要立即告知他。

这一集有大量前沿内容，对某些人来说可能信息量过大。但我要说的是，这一集真的既是给你们听的，也是给你们的代理听的。我强烈建议你们把文字稿给自己的代理看，让它根据你当前的配置、近期的工作和既定目标，判断我们讨论的哪些想法最有价值。我自己已经这样做了，我的系统因此变得更好了。

构建第二大脑

Nathan Lebenz

Daniel Miessler，欢迎再次来到 The Cognitive Revolution。

Daniel Miessler

太好了，感谢邀请。

Nathan Lebenz

从我们第一期节目到现在才四个月。时间在 AI 领域过得飞快，你比大多数人都清楚。我很期待这次对话，因为从那以后我一直在做的事情——受你和其他人的启发——就是搭建我自己的个人 AI 基础设施。到现在已经走得够远了，我觉得它对我确实很有价值，可能值得分享。但与其在这里自说自话，我觉得更有帮助的方式可能是跟你分享我做的东西，在各个维度上获取你的反馈——你觉得我做得好的是什么，可能还有哪些价值我没抓住，可能不知不觉留下了哪些安全漏洞。谁知道我们还会聊到哪些话题呢。我觉得这会很有趣。谢谢你回来做这期节目。

Daniel Miessler

没错，太棒了，迫不及待想听。

Nathan Lebenz

我想先说一下起点。上次我们谈话后——作为背景，那时候我已经用 Claude Code 做了不少事情，写各种应用什么的，但还没有认真考虑过我自己的个人 AI 基础设施到底是什么样的，没有真正打造一个体现我所有个人特色的小天地。

所以我拿了你在 GitHub 上的仓库，还有一个朋友 Chris 为我创建并分享给我的另一套工具，把它们放在我主笔记本上的两个仓库里。然后我就开始问 Claude：好的，这就是我想做的事。我想做一个自己的版本。这两个人都比我做得多，是我的灵感来源。我让 Claude 审查这两个仓库，比较和对比，问我一些关于我想做什么的问题，最终做出某种更有我个人风格的综合版。

然后我的大目标——这对我来说一直是个长期追求——是让模型能够以我的风格写作。我先声明一下：我至今仍然会对 AI 写的东西做大量编辑，从来不会直接把 Claude 的输出原封不动地发出去。实际上我只做过一次，是在跟 Anden Labs 的那期播客里，因为他们在做 AI 自主商业，所以我觉得让 Claude 完全独立写那期的开场白并照着念很合适。但除此之外，我还是会做大量编辑。不过在写播客开场白这件事上，我发现即使我坐在那里把整篇重写，它也能确保我涵盖了所有要点，至少有一个不错的初稿。

Daniel Miessler

嗯。

Nathan Lebenz

这种方式用了很长时间效果都不错。但关键是它基本上每次都是同一个流程：给它以前的 50 篇文章和当前这期的文字稿，让它写一篇新的。但我就想，这已经够好了，我打赌对于其他用途也能得到高质量的输出。但正如我们上次讨论的，上下文成了最大的瓶颈——系统是否知道这些人是谁、我和他们的关系如何、我在这类情况下通常怎么回应？答案是不知道，至少默认情况下不知道。

所以我在一月份做的第一件大事就是建立了一个本质上是”第二大脑”的系统，首要目标是：能不能把我所有的数字输出——现有的还相当多——收集到一个地方，使它可搜索、以某种方式索引、按质量筛选，从各种角度来处理？第一步就是导出。从 Gmail 导出、从 Slack 导出、所有我发过的推文、所有播客，等等。

Daniel Miessler

等一下。Gmail 里你认为什么算是内容？

Nathan Lebenz

我用的筛选条件很简单，就是只要有 from:me 的邮件。在 Gmail 里搜索 from:me，就能得到你发起或回复过的所有邮件线程，我基本上把这些全部导入了。

Daniel Miessler

好的。但我是说，这是为了追踪你和谁交流过、商业关系是什么，还是你觉得你可能在邮件里发过什么好文章或好想法想要留存下来？

Nathan Lebenz

两者都有。在后处理层面，我在此基础上尝试了几个后处理层。其中一个是——大约同一时期我和一个创业公司合作，他们想为个人微调模型。他们的想法是：我们需要创建能帮助人们保持经济杠杆的 AI，唯一的方法就是创建真正增强他们能力的东西，而不是那些大型云端巨无霸——那些公司显然更想用 AI 替代人而不是增强人。

为此他们需要写作样本。所以我有一个后处理层试图识别：我发过的哪些邮件适合做写作样本，真正展现我有思考深度的一面，而不只是日常操作性的来回沟通。效果还不错。我记得当时用了 Gemini 3 Flash 来处理大量邮件线程，在几个维度上打分——原创性、实质性等等。

在此基础上做一些查询，就能把这些内容提取出来，放进一个主写作样本文档。不管你需要五万词还是二十万词，我都有足够的素材，但关键是怎么让精华浮上来。这是一个初始用途。

后来还有另一个后处理——当时大家突然都在做 Wiki。我有个想法：既然我从大约八个不同来源收集了所有这些数据——iMessage、还下载了一个叫 Beeper 的应用用来把各平台私信整合到一起——那我可以回到过去做定期摘要。最终我定下来大约做按月摘要。对我来说平均每个月大概有二十万个 token，包括我在对话线程中的贡献和别人发给我的内容。

基本思路是创建一个涵盖我数字生活的全景图，然后开发一个提示词，把每个月二十万个 token 的内容做大约一个数量级的压缩。

Daniel Miessler

从二三十万压缩到两三万个 token，老实说这已经不少了。两万个 token 对一个月的描述能写得很细了。想想每个工作日一千个 token，你每天真的有一千个 token 那么多内容吗？还可以再压缩一次压到两千对吧，如果需要的话？

Nathan Lebenz

是的。我在年度层面做了这个。按月浏览了过去五年的数据，拿到这个月的原始数据加上前几个月的摘要，然后滚动创建月度摘要。在此之上再叠加年度摘要，然后再叠加高层级的总结——这就是我想要达到的效果，跟你搭建的东西很像：这是当前状态的全景，有非常深入的历史背景，然后是在此之上的当前事态。

后来随着 Karpathy 和其他人提出 Wiki 的概念，又加了一层：在此基础上建一个 Wiki，识别个人、回顾所有时间的数据，总结与某个人、某个组织、某些想法的关系。我想现在那个 Wiki 大概有五百多篇文章了。

Daniel Miessler

这些文章有用前置元数据标记吗？比如带有对其他文档的引用？是 Markdown 还是别的什么结构？

Nathan Lebenz

基本上就是纯文本 Markdown。这是个好问题，因为在做这个过程中需要反复迭代的精细环节之一就是如何最好地链接回源材料。我目前的方案——不确定是不是最佳方案，但效果还不错——是让摘要提示词在行内插入对源材料的引用，用独特的原文摘录。我最终要求的是大概两到二十个词的片段，如果你搜索那个确切的字符串，就能立刻定位到那个文档。基本上就是直接引用，再加一点额外元数据，比如一段引文然后标注”这来自某某人在 LinkedIn 上的私信”之类的。

总体来说效果还不错，但我肯定对改进空间持开放态度。

Daniel Miessler

有意思。我基本上也做了同样的事情。我本质上是逆向工程了 Obsidian 的功能但不用 Obsidian 客户端。我想在脑子里分清楚，大家对 Obsidian 那么狂热到底有多少是因为客户端，有多少是因为连接性。因为我不是一个重度客户端用户，我觉得客户端最终会消失，这可能也是我们会聊到的话题之一。但我想弄清楚它用的模式是什么——就是文档之间的引用关系。所以我早就开始搭建这个了，这就是我记忆文件的结构。后来 LLM Wiki 出来后，一切变得更具体了——我想”等等，我可以对所有东西都这样做”。所以我的书签也用这种方式，所有东西都是高度关联的 Markdown 文件。

我觉得我们的做法基本上是一致的。

Nathan Lebenz

我得说它在最初的冒烟测试上非常有效——说实话这些测试本身就已经很有价值了。基本就是：我隐约记得发生过什么事，有过某些通信，可能不确定是跟谁，可能不确定具体说了什么，但我能模糊地指向某个方向，知道数据就在那里。AI 能找到并取回来给我，让我回忆起上下文吗？

答案是能，效果非常好。我经常只是说”嗯，我好像知道有这么个事”，然后转眼间就是”是的，我想我找到了，是这个，这就是发生的经过”。太不可思议了，这本身就是一个令人难以置信的工具。

我还注意到它在某些方面改变了我的行为。通话录音是我以同样方式索引的另一个数据源。我一直在用 Fireflies。

Daniel Miessler

Granola 或者别的什么？

Nathan Lebenz

嗯，Granola 我们做过一期节目，他们还赞助过播客一段时间。所以我实际上两个都在用。但 Granola 有一个特性是它不录制原始音频。它有文字稿但没有源音频文件。我两个都想要。

Daniel Miessler

他们有自己的理由。我觉得他们认为在很多场景下这个要求可能太过了。他们在尝试企业部署，人们是否真的完全准备好了，这是个问题。

Nathan Lebenz

我准备好了，但我也能理解，他们试图部署的那些公司里，很多人可能更接受没有原始音频的方案。不过这确实是让我保持两条路线的一个功能差异。

说到这里，有一个我觉得很重要的概念——技术发展这么快，我们应该始终保留原始数据。

Daniel Miessler

对，我非常赞同”全量保存”这个理念。技术变化太快了，我们总是希望保留原始数据。因为在任何时刻，假设 Anthropic 出了第五代或者 OpenAI 出了第六代，是一个阶段性的飞跃。你的第一个提示词可以是”去看看我现在的系统。但你看这边——你其实有所有原始数据。让我们从头开始重建，做得更好”。然后它就会说：“好的，在那个级别我们不需要做摘要了，我们可以在完全不同的级别来做。”

但你绝不想遇到这种情况：AI 说”当然，我完全可以把这个做好一千倍，把原始数据给我”，你却说”我没有原始数据，只有旧的摘要”。

所以总结一下，我同意你的直觉。我们应该始终保留原始数据，因为只要有原始数据，我们就可以随着 AI 的进步从头重建整个系统。

Nathan Lebenz

是的，除非保留原始数据是成交的根本障碍——对我来说它不是。我确实很庆幸有过去三年的通话录音。相对于三年前的转录水平，即使在这个相对窄的领域，现在能得到比三年前 Fireflies 原生提供的好得多的转录。所以能访问所有这些原始音频文件确实很有帮助。

Daniel Miessler

对，而且我说的是整体意义上的一切——你所有的 Gmail 邮件、所有东西。也许我们应该维护一个原始存储——虽然我还没做——每次做 Gmail 解析时把所有东西放进一个仓库里，这样就不必再重复整个过程了。在任何时候你都可以说”这是原始数据，重来”。想想这适用于所有东西——你做过的每个视频的文字稿、所有邮件、所有通话、公司的所有 Slack 消息。

Nathan Lebenz

我基本上就是这么做的。也许不是把所有原始音频都保存在电脑上，但那种我可以交给新模型说”从这里开始重建”的原始数据，我都有了。现在基本上存在一个大约 1GB 的数据库里。

Daniel Miessler

这是 SQLite 吗？本地的？

Nathan Lebenz

是的，SQLite，在本地电脑上。然后我也推到了 GitHub，虽然最近因为一些安全问题我开始重新考虑这个做法。但目前所有数据都在这个超级数据库里。

对于没做过这种事的人来说——坐在那里跟 AI 说”你能不能写个工具帮我获取这个那个的访问权限”——这种体验真的很棒。即使在一月份，它通常毫无障碍就能编出工具来。

所以我现在的心智模型是：我的笔记本是给我用的，上面的 Claude Code——我也开始试用 Codex 了——我把这些视为”高访问权限、低自主性”。它们的指令基本上是：完全按我说的做。可以搜索网上信息、搜索我的个人上下文，但不要以我的身份发邮件、绝对不要冒充我、不要超出我明确告诉你做的事。

Mac Mini 上则是反过来：相对较低的访问权限，但高自主性。这引出了大量关于架构设计的问题。

安全与信任

Nathan Lebenz

在网络连接方面，我现在用 Tailscale 来创建私有 VPN。用 Screens 应用从手机登录 UI，用 Termius 通过 SSH 做命令行操作。

这个网络设置现在对我来说相当好——能从手机无缝连接到两台电脑，不管我在哪里。这很关键，因为总有东西需要重启——可能跑了好几天没问题，但某个 Telegram 通道突然连不上了之类的。如果你没法远程登录去跑个命令，你出了三天差回来发现什么都不工作就太痛苦了。

但我现在越来越注意安全了。以前觉得隐蔽即安全就够了。但在 AI 大规模监控的时代显然不够了。

从安全角度你怎么看这个设置？

Daniel Miessler

我觉得不差。我有点担心那些小型应用——你使用的那些小公司，它们方便好用，但公司越小，越不可能有专职安全人员。如果它们存了密码之类的东西，随时可能被攻破，特别是现在有各种代理在跑自动化漏洞赏金。

所以我的第一条启发式规则是：把任何敏感的东西托付给尽可能少的公司。我觉得 Tailscale 是个不错的方案。还有一个开源版本叫 Headscale，可以在 Cloudflare 上自己跑。

我喜欢不让任何东西面向互联网，因为传统的 VPN 像 IPsec 和 PPTP 都在互联网上监听端口，蠕虫或漏洞可以直接打过来。而 Tailscale 是出站连接的，技术上你没有开放任何端口。

缺点是如果 Tailscale 本身被攻破，攻击者就能在所有人的内网上横行。对我们这种人来说唯一的安全保障是：如果 Tailscale 被攻破了，他们会先打别的更大目标，希望我们能提前知道并关掉一切。但这确实是一个重大考量——你的瓶颈在哪里？你把东西给了多少家公司？总的来说，Tailscale 不是一个差的方案。

Nathan Lebenz

归根结底某种程度上还是靠隐蔽性来保安全。关于工具，我还用了 1Password——有一个家庭账户，有专门给代理共享的保险库。命令行安装在 Mac Mini 上，代理可以通过命令行访问密码。

我设了两个保险库：一个是”代理自动库”，代理可以随时自由使用，里面放的是 Brave Search API 密钥之类的东西。另一个是”需请求库”——技术上访问权限一样，但代理被要求先发消息获得批准后再使用那些密码。

对于 API 密钥，我用的是 Infisical。结构也差不多。

我不太理解这些公司声称的安全性到底怎么样。比如把所有密码放到 1Password 的云端应用里，我是变安全了还是变危险了？然后你读到他们网站上说的什么”双重超级加密保护”——我觉得挺好的，但我不太懂那到底是什么意思。你觉得我应该对这些方案有多大的信心？

Daniel Miessler

好问题，这方面确实很混乱。如果没人想攻击那个公司，还凑合着够安全。问题是有人可能会关注——因为列出一个”你应该攻击哪些应用来获取用户凭证”的清单太容易了。

而且因为我们都在公开谈论自己用的技术栈，所以攻击者确切知道该去攻击哪些公司。回到你的问题——如果有人有技术地针对某个具体目标，特别是现在有这些好模型，大多数公司的安全是不足以抵御的。特别是如果他们有员工、可以被鱼叉式钓鱼——如果你有几天、几周甚至几个月持续尝试——攻破这些公司虽然不是轻而易举，但也不算太难。

我会把你存在小公司云端的任何东西视为最终会被攻破的——问题只是多快、里面有什么。这就是为什么要限制公司数量。我尽量多用 Google 和 Apple 的东西，因为它们的安全团队很庞大，一直在监控。如果出事了，会同时影响很多人，不会首先影响我们，信号会很快传回来。

屏幕共享、密码管理，我尽量用操作系统原生的功能。还有一个很好的选择是 AWS Vault 来存储凭证。去大公司那里用那些被攻击最多、安全人员最多的产品。因为保护小公司的真的只是隐蔽性。

Nathan Lebenz

这些公司声称的”双重加密”——就算一个员工被黑了甚至变节了也拿不到用户密码——你基本上不相信？

Daniel Miessler

在很多情况下不是真的。我都经历过——做过 PCI 审计员，在 Apple 做过内部安全评估，做过安全声明，也帮过市场部把声明跟现实对齐。

通常情况是安全团队说”这是我们的目标”，市场部听完就说”那我可以说我们是端到端加密吗？“——不不不，不是端到端加密。市场部不了解真实状况，所以他们在做声明，安全领导在做声明，然后还有安全团队实际了解的真相，以及他们可能不知道的部分。这些声明在公开发布时可能是真的，然后一周半后一个小配置改动就不成立了——配置改动随时都在发生。

公司越小，这些声明越不可能是真的——他们光是把产品做出来就已经竭尽全力了，可能根本没有安全团队或者有一个擅长安全的 IT 人员。我会把小公司的这些声明视为——虽然不至于完全不可信——但可信度相当低。

Nathan Lebenz

你的私有网络用开源方案跑在 Cloudflare 上。Cloudflare 达到了你的”巨头”标准吗？

Daniel Miessler

达到了。第一，他们有很大的工程团队；第二，他们一直在被攻击。但更重要的原因是攻击面不明显——因为它跑在 Workers 里面。你得攻击 Workers 或者攻击 Cloudflare 主账户——那意味着要大规模攻破 Cloudflare。又是那种所有人都在猛推的巨大门——如果门倒了，所有人都会知道。跟 GitHub 被全面攻破一样，人们会很快知道。

我找的就是这种”门上有警报”的东西。而且 Cloudflare 里没有什么明显的位置让你攻破账户后就能找到这些东西。有多层防护让它更安全。

Nathan Lebenz

你的密码或令牌共享也用类似方式吗？

Daniel Miessler

我主要用本地文件加钥匙串（Keychain）。所有超级敏感的东西都是纯本地 Keychain。

Nathan Lebenz

让我再给你多介绍一些我的设置。有一件花了我很长时间才想通的事——AI 不是很可靠，某些情况下相当容易上当，容易受提示注入和对抗性攻击。当我想让这些低访问权限高自主性的代理出去做完整项目、通过多种交互模式跟外界打交道时，这些都是问题。

我给 Mac Mini 上的代理创建了自己的 Gmail 账户。很多账户会用我的，但它们确实有自己的 Gmail。它们不被要求隐瞒自己是 AI 的事实。它们被要求是 AI 就坦承——但也不要求主动在每次交互开头就说”嗨我是 AI”。我觉得你应该先展示一些质量和价值，然后人们发现是 AI 的时候，希望他们会被打动而不是反感。

代理被要求以 AI 身份出去跟外界打交道，用这些名字，但绝不撒谎。我还在找平衡——到底多主动地表明身份？我觉得不需要每次交互开头都说”嗨我是 AI”，应该先展示一些实际的质量和价值。

当这些代理出去做这些事时，一个我预期会越来越多出现的趋势是：人们发现对面是 AI 后，会开始故意捣乱。

所以我需要一种方式让自主代理能联系主代理或者我本人获取更多信息，但又不能给它们所有信息。它们访问权限低但不是零，上下文少但不是零。

我做的一件事是把笔记本上的 Wiki 做了一个”助理版本”——我用的启发式规则是：假装这是一个我正在入职的人类助理，做一个给他们看的版本。他们应该有联系人信息和很多东西，但如果我的通信记录里有些内容、当事人会觉得”你为什么把这个告诉你的助理？“——那就不应该放进去。

然后我搭建了一个消息总线，托管在 Mac Mini 上。任何代理——不管是我还是 Mac Mini 上的自主代理——都可以向消息服务器发消息。笔记本不断轮询它，还会给我手机推送通知。我们可以提供信息但不会太多——如果有人对抗性地劫持了我的代理，我们也能察觉”等等，你为什么问这个？”

然后还有一个问题：我们都在这个 Tailscale 网络上，怎么确保 Mac Mini 上的代理不能直接侵入笔记本？我尝试建立了一种单向层级控制——Mac Mini 不能 SSH 到笔记本，但笔记本可以 SSH 到 Mac Mini。我的手机也可以 SSH 到两台电脑。

在代码仓库层面也有层级结构——最顶层的代理在”第二大脑”仓库里，它知道一切。不同的仓库在层级下面拥有部分信息。有些仓库在两台电脑之间镜像，其他代理也有访问权限包括写入权限。它们也有自己的 GitHub，可以提交推送，但只限于特定仓库。

“第二大脑”仓库只在笔记本上，不跟代理共享。播客制作仓库是共享的。共享 Wiki 当然也是共享的。还有 Mercury 信用卡——为每个商家创建独立的限额信用卡给代理用。我愿意接受它们买错花生酱的风险，但只要控制在每周 500 美元以内。

我觉得一个大的领悟是：我最顶层的第二大脑 Claude Code 代理可以修改下面所有这些仓库，而不需要成为那些代理本身。早期我会去提示自主代理让它们自我增强、自己提交修改。现在我意识到最好让层级顶端的单个代理来修改所有需要修改的仓库，然后给代理下命令：“你有来自最高级别 AI 主管的更新，去更新自己。”

Daniel Miessler

我觉得这总体上是正确的。我听到了很多不同的仓库。我在尝试更统一或简化。我有一个 Unsupervised Learning 统一 GitHub 仓库。我用 GitHub 的原因是它已经内置了很多好用的原语——Issue、评论——而且这些回复可以被邮件触发。所以你可以用邮件或 GitHub 命令行跟文件结构或系统结构交互——基本上免费获得了一个项目管理系统。

我有独立的代理，在独立的 Mac Mini 上。它们在我防火墙内的 DMZ 里——本质上是互联网隔离箱，我把它们当成云端服务器来对待，就像真正独立的员工。它们有自己的名字、姓氏、头像、Mac 账户、Google 账户和 AI 账户。

我有三个——Devi、Saurin 和 Meera——按不同角色分工。Kai 可以看到所有代理和 GitHub 仓库。每个代理也能看到 GitHub 仓库。我可以说”有一个项目要做”，它会自动被标记为待处理。

更酷的是代理可以每五分钟检查一次仓库，看有没有新任务——谁先到或谁最合适就谁来做。看到一个未解决的 GitHub Issue，就标记为”正在处理”然后开始工作，把结果带回来。我所有的 AI 都能通过查看 GitHub 的状态来了解公司的状态。

在隔离方面，所有东西在各个独立的机器上都是分开的——包括 AI 账户。

Nathan Lebenz

所以你觉得 GitHub 可以替代我的自定义消息总线？

Daniel Miessler

大概可以。你可以用 GitHub 应用作为跟代理协作、评论、组织工作的地方。

Nathan Lebenz

Daniel Miessler

Nathan Lebenz

而且我要说，这些东西最重要的安全系统就是你必须有好的提示注入防御。这是进入你整个系统的头号入口。这也是我喜欢把爆炸半径限制在完全独立的机器上的另一个原因。

Kai 回到你另一个要点——Kai 就像我的延伸。他帮我管理整个系统，因为他有完全的访问权限。这就是零号环。它们被视为员工，所以有所有这些分离措施。但 Kai 可以在命令行上直接操作它们、SSH 进去、做任何配置修改包括更新。

Nathan Lebenz

还有一个我有的是应急响应技能。我基本上可以发一个命令，所有密钥就会轮换。

Nathan Lebenz

Daniel Miessler

Nathan Lebenz

遇到过一些。有些号码能用有些不行，有时候会被告知”这类号码不允许用于外呼”之类的。我正在看一些其他供应商，但我确实有一个 Twilio 的号码是能用的。这方面有点灰色地带，类似语音电话垃圾邮件的情况。

定时任务与持续改进

Nathan Lebenz

不过对 Kai，有一件事我经常做，这涉及到一个概念。你知道”苦涩教训工程”（Bitter Lesson Engineering）吗？

Nathan Lebenz

Daniel Miessler

我有一个技能叫”苦涩教训工程”，也是一个计划任务。核心概念源自 Richard Sutton 的苦涩教训——意思是：随着时间推移，你搭建的脚手架会变得越来越蠢，因为 AI 在变得越来越聪明。你指定给它的那些具体做法会变得越来越笨——那是我们自以为聪明的产物。所以我在持续检查：我是不是过度工程化了这个技能？是不是过度工程化了分离架构？这就是为什么我只用一个 GitHub 工作仓库——AI 越智能，我就越能精简这些。我一直在防范过度构建。

Nathan Lebenz

你就是让它提出变更建议然后你来批准？不允许它自动实施？

Daniel Miessler

不，不自动实施。我有一个”升级”技能，定期运行。系统也会提醒我很久没运行了。这个技能会查看 AI 领域最近发生的一切，加上我们算法执行中所有不顺利的地方，加上 Anthropic 发布的所有博客、工程文章和发版说明，然后跟我们的系统对比，推荐我们应该实现什么。我大概每两周运行一次。

Nathan Lebenz

我最近看到一条推特让我印象很深——基本在说如果你想让这些东西运作良好，你应该做比你直觉上认为的多 10 倍的压缩、后处理、交叉检查和维护。你提到的好几件事都属于这个范畴。还有其他维护工作或者方法吗？

Daniel Miessler

AI 意识

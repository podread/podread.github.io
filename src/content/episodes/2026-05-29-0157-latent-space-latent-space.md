对，在后台 Agent 系统里，你需要做一个决定：Agent 到底跑在哪里。通常的说法是"控制框架在箱内"还是"在箱外"（in the box vs out of the box）。

把 Agent 跑在箱内，你要做一些取舍。最大的负面取舍是安全性，因为 Agent 就跑在那个箱子里。除非你做了额外的安全设计，否则所有的密钥也得放进那个箱子里。而 AI 的行为是不可预测的，你很容易就意外地泄露了密钥，或者出现其他意想不到的行为。

箱外方案的思路是 Agent 的主体不直接跑在沙箱里，而是让 Agent 的"大脑"跑在某种工作进程或控制平面上。沙箱则充当"双手"——大脑通过工具调用来操作沙箱里的环境。

另一个取舍是，在我看来，箱外方案的复杂度要高很多，因为你需要管理外部状态。而箱内方案的所有 Agent 状态都在箱子里面，虽然你可以持久化到别的地方，但一切都是就近的，需要担心的事情更少。

我们的基础设施团队特别厉害，他们深入调查之后发现——问题的根因其实超级简单，但是一个非常细粒度的细节：很多虚拟机底层实际上用的不是真正的文件系统，而是网络文件系统——东西实际上缓存在网络上、在 S3 里面。所以你做 grep 的时候，实际上是在做网络调用。每次都是。这就是为什么这些机器上的 grep 特别慢。

这又回到了——我们为什么要做这么多疯狂的基础设施工作才能让这些机器正常运转。如果你自己尝试做这些，会有大量这样的小细节。所以我们最终不得不替换掉那个网络文件系统。

基本直觉是：假设你有 1TB 的磁盘，你的 Agent 只在上面写了 100 行代码。保存和重启这个磁盘要多长时间？大部分系统因为没有针对这种场景优化，所以工作量是 TB 级别的——你得保存所有东西然后再全部加载回来。

在我们的系统里，我们构建了一个增量式的文件系统。每次保存和重启时，工作量只跟文件系统的差异成正比。这大大缩短了 Devon 的启动时间。其实我们现在内部已经有了更新的系统了，但对，你需要把很多微小的细节都做对，才能让 Devon 的日常体验变好。
